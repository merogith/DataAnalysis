## 3. Project 1 — Pharmaceutical CRM Sales Analytics (SYNTHETIC)

**Question:** Do sales calls drive prescriptions, and which doctors should the team prioritise?

**Data** (generated by `data/generate_data.py`, seed = 42, year 2024):
| File | Rows | Key columns |
|---|---|---|
| `regions.csv` | 5 | region_id, region_name, market_size |
| `products.csv` | 5 | product_id, product_name, therapy_area, list_price_usd |
| `reps.csv` | 40 | rep_id, rep_name, region_id, tenure_years, email |
| `hcps.csv` | 600 | hcp_id, specialty, region_id, potential_score, decile |
| `calls.csv` | ~8,910 | call_id, rep_id, hcp_id, call_date, call_type, duration_min |
| `prescriptions.csv` | ~22,300 | script_id, hcp_id, product_id, month, script_count |

- **5 products:** Cardiplex (Cardiology, flagship ×1.30), Neurova (Neurology ×1.00), Pulmoair (Respiratory ×0.85), Glucosane (Endocrinology ×1.10), Dermacalm (Dermatology, niche ×0.60).
- **5 regions:** Northeast, Southeast, Midwest, Southwest, West.
- **Built-in signal (ground truth):** `scripts = baseline + responsiveness × calls + potential + product effect + noise` — so calls genuinely correlate with scripts.
- **Injected messiness (the cleaning task):** ~30 duplicate calls + ~40 duplicate scripts; mixed-case region/product names; stray whitespace; 3 mixed date formats (`%Y-%m-%d`, `%m/%d/%Y`, `%d-%b-%Y`); ~8% missing rep emails; ~6% missing potential scores; ~4% missing call durations; impossible values (negative durations, a 999-min call, potential >100, a 300-script month, a negative script count, a future date 2025-07-15).

**Methods:** cleaning log → EDA → OLS regression (statsmodels) → **KMeans (k=4)** doctor segmentation (scikit-learn) → SQL `RANK() OVER (PARTITION BY region_name ORDER BY scripts DESC)`.

**Headline numbers:**
| Metric | Value |
|---|---|
| Calls ↔ scripts correlation | r = 0.49, R² = 0.24, p ≈ 1.5e-37 |
| Lift per extra call/year | ~5.5 more scripts |
| Top product / region | Cardiplex / Southeast |
| Top vs bottom rep (scripts) | 6,007 vs 154 |
| KMeans segments (k=4) | Champions 133 · Steady mid-tier 51 · High-potential growth 99 · Low priority 317 |
| Concentration | Top 20% of doctors ≈ 37% of all scripts |

**Recommendation:** shift calls to the 99-doctor High-potential-growth segment; retain Champions; coach weak reps on top-rep patterns.

**Dashboard exports:** `fact_scripts` (22,258), `fact_region_product_monthly` (300), `dim_hcp/product/region/rep`.

## 4. Project 2 — Free-to-Play Game Economy & Balance (SYNTHETIC)

**Question:** Is the economy balanced, are items fair, and what makes players stay and pay?

**Data** (generated by `data/generate_data.py`, seed = 20260616, 6,000 players, 120-day window from 2026-01-01):
| File | Rows | Key columns |
|---|---|---|
| `players.csv` | ~6,012 | player_id, install_date, platform, country, acquisition_channel |
| `sessions.csv` | ~95,431 | player_id, day_since_install, duration_min, session_time |
| `matches.csv` | ~63,481 | match_id, player_id, match_time, item_equipped, won, match_duration_min |
| `currency_transactions.csv` | ~291,843 | txn_id, player_id, txn_time, txn_type, flow (source/sink), amount |
| `purchases.csv` | ~992 | purchase_id, player_id, purchase_time, item_name, payment_type, usd_amount |
| `items.csv` | 11 (10 + 1 dup) | item_name, slot, base_price_soft, design_power, is_premium |

- **10 items.** Designed (ground-truth) win rates: Phoenix Blade 0.61 & Aegis Shield 0.59 (both **premium → overpowered**), Rusty Dagger 0.41 (**underpowered trap**); the rest ~0.49–0.52 (balanced).
- **Built-in stories:** mild **inflation** (faucets out-mint sinks, +0.25%/day faucet creep); **D1 retention cliff**; payers are a small high-propensity minority.
- **Platforms:** iOS / Android / Web. **Countries:** US, BR, DE, IN, GB, JP, FR, TR. **Channels:** organic, paid_social, influencer, cross_promo.
- **Injected messiness:** duplicate rows; mixed-case + whitespace platform/item names; two timestamp formats (ISO + US M/D/Y AM/PM); negative durations; ~1.5–3% missing values.

**Methods:** cleaning → EDA → faucet-vs-sink net flow → **proportion z-test + 95% CIs vs 50%** for item balance (statsmodels) → D1/D7/D30 cohort retention → monetization funnel → **LogisticRegression** payer model on first-3-day behaviour (scikit-learn; accuracy, AUC, odds ratios) → SQL `SUM() OVER (PARTITION BY player_id ORDER BY ...)` running wallet + `RANK()` item win-rate ranking.

**Headline numbers:**
| Metric | Value |
|---|---|
| Net currency flow | +6.8M coins (inflation); sinks recover only ~63% of coins minted |
| Overpowered items | Phoenix Blade 61.3%, Aegis Shield 59.4% (both premium → pay-to-win) |
| Underpowered item | Rusty Dagger 41.8% |
| Retention | D1 47% · D7 18% · D30 4% |
| Payer rate / ARPPU | 5.8% / $15.61 |
| Payer model | AUC 0.80, ~74% accuracy (first-3-day behaviour only) |
| Win rate premium vs free | 60.3% vs 49.8% |

**Recommendation:** add a coin sink (target net ~0); nerf the two OP premium items + buff Rusty Dagger; fix onboarding to attack the D1 drop; target a fair early offer at high-propensity players.

**Dashboard exports:** `fact_daily_economy` (210), `dim_item` (10, with z-test verdicts), `dim_player` (6,000), `dim_date` (210).

## 5. Project 3 — US Stock Market: Trends, Risk & Valuation (REAL DATA — flagship)

**Question:** Where does the market stand, how risky is it, is it expensive, and does that predict future returns?

**Data** (downloaded by `data/download_data.py`; real public sources via GitHub `datasets/` mirrors of Shiller, CBOE, LBMA — same series as yfinance/FRED). Falls back to clearly-labelled synthetic data if the network is blocked (leaves a `_SYNTHETIC_FALLBACK_USED` marker).
| File | Rows | Source / content |
|---|---|---|
| `raw_sp500.csv` | 1,865 | **Shiller** S&P 500 monthly since 1871: price, dividend, earnings, CPI, 10y rate, real values, PE10/CAPE |
| `raw_vix.csv` | 9,206 | **CBOE VIX** daily since 1990 (fear gauge) |
| `raw_gold.csv` | 2,321 | **LBMA** gold monthly |
| `raw_constituents.csv` | 503 | Current S&P 500 companies + GICS sector |

- **Key cleaning decision:** recent months store `0.0` as a placeholder for not-yet-reported fundamentals → converted to missing (NaN) so averages aren't distorted.
- **Methods:** real-trend (log scale), CAPE history, return distribution, VIX → drawdown ("underwater") view → **OLS regression of CAPE vs the next 10-year real return** (statsmodels) → cross-asset correlation matrix → SQL `LAG()` window function to rank best/worst calendar years.

**Headline numbers:**
| Metric | Value |
|---|---|
| Long-run real price growth | ~2.4% per year |
| Current CAPE | 30.8 — higher than 95% of all months since 1881 |
| CAPE → forward 10y return | Negative slope, R² = 0.11, p ≈ 1e-40 |
| Implied next-10y real return | Below the 2.3% historical average |
| Worst real drawdown | −81% (trough June 1932) |
| Stocks vs gold correlation | ≈ 0 |
| Worst / best calendar years | 1931 (−46%) / 1933 (+46%) |

**Recommendation:** set return expectations to "below average" given today's valuation; diversify with low-correlation assets (gold/bonds); automate contributions/rebalancing to avoid selling in downturns.

**Dashboard exports:** `fact_market_monthly` (1,865), `dim_year` (156), `dim_sector` (11).

## 6. The SQL (one window-function query per project)

- **P1 — rank reps within their region:**
  `RANK() OVER (PARTITION BY region_name ORDER BY scripts DESC)` plus `AVG(scripts) OVER (PARTITION BY region_name)`.
- **P2 — running wallet balance per player + item ranking:**
  `SUM(amount) OVER (PARTITION BY player_id ORDER BY txn_time)` and `RANK() OVER (ORDER BY win_pct DESC)`.
- **P3 — best/worst calendar years:**
  `LAG(dec_close) OVER (ORDER BY year)` to compute each year's return, then rank.

All run in an **in-memory SQLite** database (`sqlite3.connect(":memory:")`, `to_sql`, `read_sql`).

## 8. Quick Q&A cheat sheet

- **Which uses real data?** Only Project 3 (Shiller / CBOE / LBMA). P1 and P2 are 100% synthetic from seeded scripts.
- **Why synthetic for P1/P2?** Real pharma CRM and game telemetry are proprietary and privacy-sensitive.
- **Are the results reproducible?** Yes — fixed seeds (P1=42, P2=20260616), and notebooks are rebuilt from `build_notebook.py`.
- **What ML is used?** KMeans segmentation (P1) and logistic-regression payer prediction (P2).
- **What database?** In-memory SQLite, demonstrating SQL window functions in all three.
- **How many charts total?** 21 (P1: 7, P2: 8, P3: 6).
- **How do I run a project?** `cd <project>`, run its `data/` script, `python build_notebook.py`, then `jupyter nbconvert --to notebook --execute --inplace analysis.ipynb` (or just open the notebook and Restart & Run All — outputs are already saved).
- **What's the BI story?** Each project exports a star schema (fact + dim CSVs) with a Tableau/Power BI build guide.
