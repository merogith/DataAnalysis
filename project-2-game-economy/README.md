# Project 2 — Free-to-Play Game Economy & Balance Analytics 🎮
**Portfolio project · SYNTHETIC telemetry · Python + statistics + SQL**

## Executive summary (for a non-technical reader)
- Built an end-to-end **F2P game-economy analytics** pipeline on realistic synthetic telemetry — players, sessions, matches, a soft-currency ledger and purchases — to answer the three questions a producer cares about: *is the economy healthy, are the items fair, and what makes players stay and pay?*
- **The economy is mildly inflationary.** Currency **faucets** (rewards) out-mint **sinks** (spending): sinks recover only **~63%** of every coin created, so wallets steadily swell — left alone, soft currency slowly loses value.
- **Two premium items are statistically overpowered.** *Phoenix Blade* (**61.3%** win rate) and *Aegis Shield* (**59.4%**) win far above the fair 50% line (confidence intervals entirely above 50%, p ≈ 0). Because both are **paid** items, the game currently reads as mildly **pay-to-win**. *Rusty Dagger* is an **underpowered "trap"** at **41.8%**.
- **There's a textbook Day-1 retention cliff:** **D1 47%, D7 18%, D30 4%** — about **half** of installs never return on day 1, but survivors are sticky. The biggest lever is the **first session**, not the late game.
- **Monetization is small but predictable.** Only **5.8%** of players ever pay (ARPPU **$15.61**), and they are sharply distinguished by **early engagement**. A logistic model predicts payers from first-3-day behaviour with **AUC 0.80** (74% accuracy).
- **Bottom line:** add a coin sink, nerf the two pay-to-win items, fix onboarding to attack the D1 cliff, and target a fair early offer at high-propensity players.

> **Data label: SYNTHETIC.** All telemetry is generated by one seeded script (`data/generate_data.py`). A public Steam/Kaggle game dataset was not reachable from this sandbox, so a reproducible synthetic generator is the standard path here (see `NOTES.md`).

## Problem
A game **producer** needs an evidence-based read on the live game's health — not gut feel — across three areas:
1. **Economy balance** — do currency sources (faucets) match sinks (drains), or is there inflation/deflation?
2. **Item balance** — are any items over- or under-powered (unfair win rates), and is the game pay-to-win?
3. **Retention & monetization** — what drives players to stay (D1/D7/D30) and to pay, and can we predict payers early?

## Approach
1. **Generated reproducible synthetic telemetry** (`data/generate_data.py`) with deliberate real-world messiness — missing values, duplicate rows, mixed-case names, mixed timestamp formats, stray whitespace, impossible negatives.
2. **Cleaned & documented every decision** in a table — de-duplication, platform/item-name standardisation, parsing two timestamp formats with one parser, turning impossible negative durations into missing.
3. **EDA** — player mix and the DAU pulse of the game.
4. **Deeper analysis** — faucet-vs-sink **net currency flow**; an **item-balance table** with a **proportion z-test** & 95% CIs vs 50%; **cohort retention** curves; a **monetization funnel**; spenders vs non-spenders; and a **logistic-regression** payer model on early behaviour (accuracy + AUC + odds ratios).
5. **SQL (SQLite)** — a `SUM() OVER (PARTITION BY ...)` **window function** for a running wallet balance per player, plus `RANK()` to rank items by win rate.
6. **BI-ready star-schema exports** + a click-by-click Tableau/Power BI guide.

## Recommendation
- **Re-balance the economy:** add/deepen a coin **sink** (cosmetics, repairs, a soft-currency battle pass) so sinks recover ~90–100% of faucets, and trim the late-game faucet creep. Target net daily flow near zero.
- **Nerf the two overpowered premium items** toward 50% to kill the pay-to-win perception, and **buff Rusty Dagger** so the starter weapon isn't a trap.
- **Attack the D1 cliff first** — better onboarding, a guaranteed early win, a D1 login reward. A few points of D1 cascade into every later metric.
- **Target likely payers early** using the model — a well-timed, *fair* first offer to high-propensity players in their first days.

## What I'd do next
- Design an **A/B test** for the sink + onboarding changes with explicit guardrail metrics.
- Move from "ever pays" to **predicted LTV** and a **survival model** for churn timing.
- Track item balance **over time** with a win-rate control chart so new content can't silently break balance.
